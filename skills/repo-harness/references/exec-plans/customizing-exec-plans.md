# Customizing ExecPlans for Your Repository

This document tells you, the LLM, how to adapt the generic `create-plan-file.md` for a specific repository. The generic file is meant to live at `docs/exec-plans/create-plan-file.md` in the target repo. It contains the process, requirements, formatting rules, and skeleton. Your job is to add a **repository-specific section** to that file so that plans written in this repo include the right tech context, commands, and conventions automatically.

## When to do this

When a user drops `create-plan-file.md` into a new repository under `docs/exec-plans/` and asks you to make it project-specific, or when you detect that the file has no project-specific section and the repo has an `AGENTS.md` or equivalent.

## What to produce

Add a single section called `## Project-Specific Conventions` to `create-plan-file.md`, placed **immediately before** the `## Formatting` section. This keeps all repo-specific content in one place and clearly separated from the portable process rules.

You also need to update three parts of the skeleton template to match the repo:
1. The `Reference:` line
2. The `Context and Orientation` checklist
3. The `Concrete Steps` and `Validation and Acceptance` example commands

## Step-by-step instructions

### 1. Gather context from the repo

Read the following files if they exist (in priority order):
- `AGENTS.md` (root) -- architectural conventions, coding standards, glossary
- `CLAUDE.md` or `.claude/CLAUDE.md` -- agent-specific instructions
- `package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod` -- package manager, scripts, tech stack
- App-specific `AGENTS.md` files (e.g., `apps/*/AGENTS.md`, `src/*/AGENTS.md`)
- `README.md` -- project overview, setup instructions

From these, extract:
- **Package manager and how to invoke it** (npm, pnpm, yarn, bun, cargo, go, pip, etc.)
- **Key commands**: dev server, type check, lint, test, build, database migrations
- **Tech stack**: framework, language, ORM, API layer, UI library, background jobs
- **Local dev URLs/ports** if applicable
- **Coding conventions** that affect plan writing (error handling patterns, file naming, DI approach, etc.)
- **Monorepo structure** if applicable (which apps exist, where plans go per app)
- **Plan directory paths** if the repo uses app-specific plan directories

### 2. Write the `## Project-Specific Conventions` section

Structure it with these subsections. Omit any that don't apply.

```markdown
## Project-Specific Conventions

### Plan Directories

> Where plans go in this repo. Override the generic guidance from the header.

Example for a monorepo:
- **App A work**: `apps/a/docs/exec-plans/active/<timestamp>-<plan-name>.md`
- **App B work**: `apps/b/docs/exec-plans/active/<timestamp>-<plan-name>.md`
- **Cross-app/shared work**: `docs/exec-plans/active/<timestamp>-<plan-name>.md` (root)

Example for a single-app repo:
- `docs/exec-plans/active/<timestamp>-<plan-name>.md`

### Tech Stack Reference

A short bullet list. Only include what a plan author needs to know to write
correct commands and make correct technology choices.

- **Language**: TypeScript 5.x
- **Package Manager**: pnpm (use `pnpm` not npm/yarn/bun)
- **Framework**: Next.js 15
- ...etc.

### Local Development

Ports, URLs, and how to start the dev server. Only if applicable.

- **Web app**: localhost:3000
- **API server**: localhost:4000

### Common Commands

Indented code block with the commands a plan author will reference most.
Group by category (Development, Database, Testing, Scripts).

    # Development
    pnpm dev                              # Start dev server
    pnpm typecheck                        # Type check
    pnpm lint                             # Lint

    # Testing
    pnpm test                             # Run all tests
    pnpm test src/path/to/file.test.ts    # Run specific test

    # Database (if applicable)
    pnpm db:migrate --name <name>         # Create migration
    pnpm db:studio                        # Open DB GUI

### Code Patterns

Short bullet list of conventions that directly affect how plans are written.
Only include patterns that a plan author needs to follow -- skip anything
already covered in AGENTS.md that doesn't change how plans look.

- Functions with 2+ params use object signatures
- Services return Result<T, E>, only throw at API boundary
- ...etc.
```

### 3. Update the header note

Replace the generic header note with repo-specific plan directory paths:

```markdown
> **DO NOT EDIT THIS FILE**
> This file is the ExecPlan template and guide only.
> Create plans in the appropriate directory:
> - **App A work**: `apps/a/docs/exec-plans/active/<timestamp>-<plan-name>.md`
> - **App B work**: `apps/b/docs/exec-plans/active/<timestamp>-<plan-name>.md`
> - **Cross-app/shared work**: `docs/exec-plans/active/<timestamp>-<plan-name>.md` (root)
```

For single-app repos, simplify to:

```markdown
> **DO NOT EDIT THIS FILE**
> This file is the ExecPlan template and guide only.
> Create plans in `docs/exec-plans/active/<timestamp>-<plan-name>.md`.
```

### 4. Update the skeleton template

In the skeleton's `Reference:` line, name the actual AGENTS.md files:

```markdown
Reference: This plan follows conventions from AGENTS.md (root),
apps/web/AGENTS.md, and docs/exec-plans/create-plan-file.md.
```

In the skeleton's `Context and Orientation` section, replace the generic checklist with repo-specific items. Example for a monorepo:

```markdown
## Context and Orientation

...

For this monorepo, always mention:
- Which app(s) are affected (web, api, admin)
- Which packages are involved (packages/logger, packages/db, etc.)
- Key domain modules touched (src/modules/billing, src/services/*, etc.)
- Any external integrations involved (Stripe, SendGrid, etc.)
- Any background jobs or queues affected
```

In the skeleton's `Concrete Steps` and `Validation and Acceptance` sections, replace the generic `npm` commands with the actual commands from this repo:

```markdown
## Concrete Steps

    pnpm typecheck
    # Expected: No errors

    pnpm test:web src/services/billing/charge.test.ts
    # Expected: All tests pass

## Validation and Acceptance

    pnpm dev:web
    # Navigate to localhost:3000/billing, observe [behavior]

    pnpm typecheck        # No type errors
    pnpm lint             # No lint errors
    pnpm test:web         # All tests pass
```

### 5. Update the Plan Lifecycle directory structure

Replace the generic structure with the actual repo layout:

```markdown
    apps/web/docs/exec-plans/      # Web app plans
    ├── active/
    ├── completed/
    └── abandoned/

    docs/exec-plans/               # Cross-app/shared plans
    ├── active/
    ├── completed/
    └── abandoned/
```

## What NOT to do

- **Don't duplicate AGENTS.md wholesale.** The Project-Specific Conventions section should contain only what affects plan authoring. A plan author doesn't need to know every lint rule -- just the commands to run and the patterns to follow.
- **Don't change the process, requirements, formatting, or guidelines sections.** Those are portable and repo-agnostic.
- **Don't remove the generic examples.** Replace or supplement them with repo-specific ones but keep them illustrative rather than exhaustive.
- **Don't add project-specific content outside the designated section** (and the skeleton updates). One section plus skeleton tweaks -- that's it.

## Checklist

After customizing, verify:
- [ ] `## Project-Specific Conventions` section exists before `## Formatting`
- [ ] Header note reflects actual plan directory paths
- [ ] Tech stack, commands, and patterns match the repo's AGENTS.md
- [ ] Skeleton `Reference:` line names the actual AGENTS.md files
- [ ] Skeleton `Context and Orientation` has repo-specific checklist items
- [ ] Skeleton `Concrete Steps` and `Validation` use the repo's actual commands
- [ ] Plan Lifecycle directory structure matches the repo layout
- [ ] No process/requirements/guidelines sections were modified
- [ ] The file still reads correctly end-to-end for someone with zero repo context
